A role-based web application for managing student records, course enrolments, and academic grades — built with Bootstrap, JavaScript, and a MySQL-backed data layer with strict access controls per user role.

The Problem

Academic institutions frequently rely on spreadsheets or siloed tools for student data — a fragile approach that breaks down at scale and lacks auditability. A centralised, role-differentiated system provides data integrity, controlled access, and a clear audit trail for records changes.

  • Spreadsheets offer no access control — any user can modify any record
  • No validation layer means corrupt data (duplicate IDs, invalid grades) accumulates silently
  • Reporting requires manual aggregation rather than automated queries

Key Features

1

Role-Based Access Control (RBAC)

Three roles with distinct permissions: Admin has full CRUD across all entities; Faculty can enter and edit grades and view enrolments; Students have read-only access to their own records. Role enforcement is server-side — client-side UI changes are cosmetic only.

2

Student Records Module

Create, update, and soft-delete student profiles with validation on all fields (unique student ID, email format, valid cohort year). Audit log records who changed what and when.

3

Enrolment Management

Course catalogue with capacity tracking. Add/drop workflow with confirmation step and waitlist for over-capacity courses. Prevents duplicate enrolments via unique constraint.

4

Grade Book

Faculty enter, edit, and submit grades per course per student. Automatic GPA calculation using weighted credit-hour formula. Grade submission is locked after the semester deadline.

5

Reporting & Export

Cohort-level and course-level summary tables with grade distribution charts. One-click CSV export via Blob API — no server-side file generation required.

Key Learnings

  • Schema-first design — normalising to 3NF before writing a single line of application code prevented data anomalies and made the query layer clean.
  • Server-side role enforcement is mandatory — hiding UI elements client-side is UX, not security. Every restricted action must be validated on the server.
  • Soft deletes over hard deletes — academic records must be archivable, not permanently removed; soft delete with an archived flag preserves data integrity.
  • CSV export via Blob API is a practical zero-dependency reporting solution for moderate data volumes.
